Build succeeded. Here's a summary of all the fixes that went into this deploy:

Why the bulk scan wasn't working:

The query used a bulk_scan_checked_at column that was only added to the Replit dev DB, not your production DB on Render. Every time you clicked "Scan All Live," the query threw a "column does not exist" error, silently fell into the catch block, set bulkScanRunning = false, and the frontend saw "scan finished" immediately — nothing was actually checked.
There was also a race condition: bulkScanRunning was set to true inside the async background task instead of before launching it, so the status endpoint could flash false in that tiny window.
What was fixed:

Removed the bulk_scan_checked_at column entirely — replaced with cursor-style ID pagination (WHERE id > lastId ORDER BY id ASC LIMIT 30). No schema dependency, works on any database as-is.
Fixed the race condition — bulkScanRunning = true is now set synchronously before the fire-and-forget, so the status endpoint always sees the correct state.
Scan now covers ALL accepted live submissions — removed the moved_to_available = 0 filter from the scan query so it also catches submissions that were paid out but never had their live_status badge updated. Reversals still only fire for unpaid ones.
Also stamps last_checked_at even when status hasn't changed, so you get a record of when each row was last confirmed live.
To pick these up on your Render deployment: push the new build (pnpm --filter @workspace/api-server run build is already done), then redeploy. The next "Scan All Live" will work through every live accepted submission in batches of 30.
